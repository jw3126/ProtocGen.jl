# Known conformance failures for ProtoBufDescriptors v1, against the Google
# protobuf conformance_test_runner from protobuf v25.9.
#
# Consumed by `conformance_test_runner --failure_list`. Each uncommented
# line is a `Required.*` test we currently fail; the runner treats them
# as expected failures so the suite reports success when only these are
# failing (and a *new* failure shows up as unexpected, which is the
# regression signal CI cares about).
#
# Categories (278 total — 188 binary inherited from Phase 9, +91 JSON
# added in Phase 12d, −1 fixed by the proto3-explicit-optional
# presence-on-default-value JSON encode fix):
#
#   * Binary (Phase 9 baseline, 188 cases):
#     - 111 ValidDataMap.*: empty entry handling (MissingDefault),
#       duplicate key/value within an entry (DuplicateKey/Value), and
#       value-before-key field order (Unordered) — all rejected by our
#       codec's strict map-entry decoder.
#     -  17 ValidDataScalar.*: BOOL with a varint payload >1 not
#       accepted as `true` (spec: any non-zero is true); proto2 ENUM
#       zero-default drop; UINT32 wire edge cases.
#     -  ~30 PrematureEof* / IllegalZeroFieldNum_*: malformed inputs
#       the spec wants rejected — we silently accept and skip.
#     -  rest: RepeatedScalar*, UnknownVarint, ValidDataOneof tail.
#
#   * JSON (Phase 12d, 91 new cases):
#     - ~70 JsonOutput mirrors of the binary failures above (same
#       root-cause map / scalar bugs, just routed through the JSON
#       output path).
#     - ~15 JSON-input edge cases not exercised by the binary path:
#       Int32FieldLeadingSpace / TrailingSpace (whitespace inside
#       number-as-string strings), Float/Double FieldTooLarge / TooSmall
#       (overflow handling), Duration / Timestamp Json input
#       Too{Large,Small}, OneofFieldDuplicate, ValueAcceptNull,
#       AnyNested, RepeatedFieldWrongElementType, EnumFieldUnknownValue.
#     - (StoresDefaultPrimitive.Validator was here pre-fix — passes now
#       that explicit-optional fields with a default value emit on JSON.)
#
# The underlying gaps stayed the same:
#
#   (a) Lenient parser. Our codec accepts several malformed-input cases
#       the spec wants rejected (premature EOF, illegal zero field
#       number, oversized BOOL varints). Same gap shows up as both
#       "Should have failed to parse, but didn't" (binary) and "JSON
#       parse should have rejected …" (JSON).
#
#   (b) Map-entry edge cases. The synthetic *Entry message decoder
#       asserts position-equality at end-of-entry and EOFs on a
#       zero-length entry. Spec allows missing key/value (use
#       defaults), duplicate fields (last-wins), arbitrary field order.
#
#   (c) JSON edge-case strictness. Whitespace inside number-as-string
#       fields, overflow detection, and a few WKT input boundaries
#       (Duration / Timestamp range) need explicit rejection paths in
#       the JSON walker.
#
# All three are real spec-conformance bugs; Phase 9 + 12d allowlist
# them so the harness lands green and any *new* failure surfaces as a
# regression. They are tracked for a follow-up codec pass.
#
